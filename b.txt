==========================

This is actually a **very good low-level question**, and asking about this in the architecture discussion will make you look practical. Since the service is **async and enterprise**, the API design will usually follow a **job-based REST pattern**.

Below is a **typical endpoint design for an async document AI extraction service**. This is very close to what teams implement in production systems.

---

# 1️⃣ Submit Document for Processing

### Endpoint

```http
POST /v1/documents/extract
```

### Purpose

Upload a document and start processing.

Since the service is **async**, this endpoint will **not return extraction results immediately**.

---

### Request

Content-Type: `multipart/form-data`

Parameters:

| Parameter     | Type   | Required | Description                      |
| ------------- | ------ | -------- | -------------------------------- |
| file          | file   | Yes      | Invoice PDF/image                |
| documentType  | string | No       | invoice, receipt, statement      |
| correlationId | string | No       | client tracking id               |
| callbackUrl   | string | No       | webhook URL for async completion |
| options       | json   | No       | extraction options               |

Example request:

```bash
POST /v1/documents/extract
Content-Type: multipart/form-data

file=@invoice.pdf
documentType=invoice
correlationId=ABC123
```

---

### Response

```json
{
  "jobId": "job_9847234",
  "status": "submitted",
  "message": "Document processing started"
}
```

---

# 2️⃣ Get Job Status

### Endpoint

```http
GET /v1/documents/jobs/{jobId}
```

### Purpose

Check processing status.

---

### Response (Processing)

```json
{
  "jobId": "job_9847234",
  "status": "processing",
  "submittedAt": "2025-02-01T10:22:11Z"
}
```

---

### Response (Completed)

```json
{
  "jobId": "job_9847234",
  "status": "completed",
  "resultUrl": "/v1/documents/jobs/job_9847234/result"
}
```

============================================================
The 5 biggest mistakes 
=======================
Great. Knowing the **common mistakes in document AI systems** will help you ask smart questions during the architecture discussion. These are issues that even experienced teams often overlook.

Below are the **5 biggest mistakes in document AI pipeline design**, and how you can spot them in tomorrow’s discussion.

---

# 1️⃣ Treating the Pipeline as a Single Service

### The mistake

Some teams build everything in one service:

```text
Upload → OCR → Classification → Extraction → Response
```

inside a single synchronous process.

This creates problems:

* slow responses
* hard to scale
* difficult to debug
* failures restart entire pipeline

### What a good architecture looks like

```text
API → Queue → Worker → Pipeline stages
```

Each stage can scale independently.

### Smart question you can ask

> Are we separating the API layer from the document processing workers so the system can scale independently?

This shows **scalability thinking**.

---

# 2️⃣ No Clear Job Lifecycle

Async systems need **clear job states**.

Bad systems only track:

```text
processing
done
```

Good systems track:

```text
submitted
queued
processing
completed
failed
cancelled
```

Without this:

* debugging becomes difficult
* clients can't understand job state

### Smart question

> Are we defining a job lifecycle state model for the async processing pipeline?

---

# 3️⃣ Not Separating Model Invocation

Bad design:

```text
Pipeline directly calls Llama or Cohere
```

This tightly couples the system to a model.

Better design:

```text
Pipeline → Model Service → Model
```

This allows:

* switching models
* versioning models
* A/B testing

Which is important because they already said:

```
Llama → Cohere later
```

### Smart question

> Will the model invocation be abstracted behind a service layer so we can swap models without changing the pipeline?

---

# 4️⃣ No Schema Standardization

This is a **huge problem in enterprise AI platforms**.

If each team returns different JSON formats:

```json
Team A:
{ "invoice_number": "123" }

Team B:
{ "invoiceNo": "123" }
```

Then integration becomes messy.

Better:

Define **standard schema**.

Example:

```json
{
  "documentType": "invoice",
  "fields": {
    "invoiceNumber": { "value": "123", "confidence": 0.95 },
    "vendorName": { "value": "ABC Ltd", "confidence": 0.92 }
  }
}
```

### Smart question

> Are we planning to standardize the extraction response schema across document types?

This is **very strong platform thinking**.

---

# 5️⃣ Ignoring Observability

AI pipelines fail often.

You need visibility into:

* OCR errors
* classification mistakes
* model failures
* slow jobs

Bad systems only log errors.

Good systems track:

```text
processing time
pipeline stage metrics
model latency
job failures
```

### Smart question

> Are we planning to add observability around each pipeline stage so we can monitor performance and failures?

---

# BONUS Issue (Very Common)

### Not handling large documents

Invoices can be:

* multi-page
* scanned
* low quality
* rotated

If pipeline isn't prepared, performance degrades.

Smart question:

> Do we have limits or strategies for handling large or multi-page documents?
